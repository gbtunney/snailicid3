# Commitlint & GitHub Label Taxonomy

## Goals

- Align GitHub labels with commitlint conventions where it makes sense.
- Keep labels understandable to humans.
- Support future GitHub Actions automation.
- Avoid overlapping label categories.
- Make issue templates, projects, labels, commits, and PRs use the same vocabulary.

---

## Labels

### Type Labels

`type:*` labels describe the kind of work being performed.

#### Mapping: Commitlint Types to GitHub Labels

The following table maps Commitlint types to their corresponding GitHub labels. These labels are
optimized for project management while maintaining alignment with Commitlint conventions.

| **Commitlint Type** | **GitHub Label** | **Description**                                          | **Typical Trigger**                 |
| ------------------- | ---------------- | -------------------------------------------------------- | ----------------------------------- |
| `feat`              | `type:feature`   | New functionality                                        | Manual                              |
| `fix`               | `type:bug`       | Bug fix or defect correction                             | Manual                              |
| `docs`              | `type:docs`      | Documentation-only changes                               | Manual                              |
| `refactor`          | `type:refactor`  | Internal restructuring without intended behavior changes | Manual                              |
| `test`              | `type:test`      | Tests or test infrastructure                             | Manual                              |
| `build`             | `type:build`     | Build tooling, packaging, workspace configuration        | Manual                              |
| `ci`                | `type:ci`        | CI/CD, GitHub Actions, automation                        | Manual                              |
| `chore`             | `type:chore`     | Repository maintenance                                   | Manual                              |
| `revert`            | `type:revert`    | Revert previous changes                                  | Manual                              |
| `changeset`         | `type:changeset` | Changeset metadata and pending version bumps             | `pnpm changeset:create`             |
| `release`           | `type:release`   | Release commits, version PRs, publishing, and tagging    | Changesets Release / GitHub Actions |

#### Notes

- GitHub labels are optimized for project management rather than strict adherence to Conventional
  Commit terminology.
- `feat` maps to `type:feature`, and `fix` maps to `type:bug`.
- `changeset` and `release` are valid commit types in this repository, even though they are commonly
  generated by tooling:
  - `changeset` represents pending release metadata.
  - `release` represents the actual release/versioning/publishing process.

### Scope Labels

`scope:*` labels indicate the specific area of the codebase that a change affects. These labels are
derived from a combination of **package names** (by removing the namespace) and special
**repository-level** scopes for broader changes.

#### Package Scopes

Each package in the monorepo should have a corresponding `scope:*` label. The `scope` is derived
from the package name by stripping the namespace. For example, the package `@snailicid3/config`
would use the scope `config`, resulting in the label `scope:config`. **Additional** packages should
follow the same pattern.

Examples:

| Namespace Package Name     | Scope          | Label                |
| -------------------------- | -------------- | -------------------- |
| `@snailicid3/config`       | `config`       | `scope:config`       |
| `@snailicid3/build-config` | `build-config` | `scope:build-config` |
| `@snailicid3/cli-app`      | `cli-app`      | `scope:cli-app`      |
| `@snailicid3/color`        | `color`        | `scope:color`        |
| `@snailicid3/logger`       | `logger`       | `scope:logger`       |
| `@snailicid3/node-utils`   | `node-utils`   | `scope:node-utils`   |
| `@snailicid3/types`        | `types`        | `scope:types`        |
| `@snailicid3/utils`        | `utils`        | `scope:utils`        |

#### Repository Scopes

In addition to **package-specific scopes**, there are special repository-level scopes for broader
changes that do not affect a single package.

| Scope     | Label           | Description                                       |
| --------- | --------------- | ------------------------------------------------- |
| `root`    | `scope:root`    | Repository-level configuration and infrastructure |
| `actions` | `scope:actions` | GitHub Actions workflows and custom actions       |

##### Notes

- The `root` scope loosely tied to the top level package in a monorepo. It represents
  repository-level configuration and infrastructure changes that affect the entire repo.

- The `actions` scope is used for changes that involve GitHub Actions workflows or automation.
  Anything in the .github folder.

### Status Labels

Use sparingly.

| Label                   | Description                              |
| ----------------------- | ---------------------------------------- |
| `status:wip`            | Work in progress                         |
| `status:blocked`        | Waiting on dependency or external factor |
| `status:needs-decision` | Requires architecture or design decision |

## Label Rules

### Use `type:*` for work classification

Examples:

- `type:feature`
- `type:bug`
- `type:docs`
- `type:refactor`
- `type:test`
- `type:build`
- `type:ci`
- `type:chore`
- `type:revert`
- `type:changeset`
- `type:release`

### Use `scope:*` for affected package or commit scope

Examples:

- `scope:root`
- `scope:actions`
- `scope:config`
- `scope:logger`

### Avoid

Avoid introducing additional prefixes unless there is a clear need:

- `category:*`
- miscellaneous duplicate labels
- labels that overlap with existing `type:*` or `scope:*`
